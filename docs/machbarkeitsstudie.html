Executive Summary

Die Kernstrategie in einem Satz

Fordify wächst in drei Stufen: Werkzeug (beste Forderungsaufstellung im Browser) → Produkt (Freemium-SaaS für Kanzleien) → Plattform (Marktplatz für Forderungsabtretungen) — wobei jede Stufe wirtschaftlich eigenständig tragfähig sein muss.

60 Mrd. NPL-Markt 2026 Deutschland, wachsend
~2–3 Wo. Architektur-Umbau Free/Paid MVP
29 €/Mo. Solo-Tier Preis 249 €/Jahr · Paddle MoR

Die zehn wichtigsten Erkenntnisse

  1. Markt groß, Lücke real. Der deutsche NPL-Markt wächst auf 60 Mrd. EUR (2026). Das Kleinforderungssegment (< 25.000 EUR) ist von keiner bestehenden Plattform adressiert.
  2. Richtiges Free Tier: „Kein Speichern + kein Export" — nicht ein Limit auf N Fälle. Erzeugt Upgrade-Druck genau im Moment des höchsten Nutzens.
  3. Paddle statt Stripe — als Merchant of Record übernimmt Paddle alle USt-Compliance-Pflichten. Spart 10–20 Stunden/Monat Steueradministration.
  4. Architektur-Umbau in 2–3 Wochen via Storage-Abstraktionsschicht. Nur ~4 Funktionen brauchen Feature-Gates.
  5. Marktplatz rechtlich machbar, aber komplex. Direkter Launch ohne Rechtsberatung wäre fahrlässig.
  6. Fordify wird DSGVO-Verantwortlicher mit Account-Modell. Supabase RLS muss vor Launch korrekt konfiguriert sein; AVV mit Kanzleien ist Pflicht.
  7. Marktplatz startet als Schwarzes Brett (Modell A) — kein Matching, keine Zahlungsabwicklung, minimales regulatorisches Risiko.
  8. Preise: Solo 29 €/Monat (249 €/Jahr), Kanzlei 79 €/Monat (669 €/Jahr).
  9. 5.000 € MRR-Ziel bei 4 % Conversion erreichbar mit ~3.500 Free-Nutzern — realistische 18-Monats-Perspektive.
  10. Alleinstellungsmerkmal: Kein Anbieter verbindet § 367 BGB-konforme Berechnung und Forderungsverkauf in einem Workflow.
⚠️

Sofort-Empfehlung

Den vollständigen Marktplatz nicht sofort bauen — zuerst regulatorische Klärung, dann Freemium-Launch, dann Marktplatz als Feature des Paid-Tiers. Parallel: Anwaltliche Prüfung (RDG, KWG, DSGVO) beauftragen.

Rechtsanalyse

ℹ️

Hinweis

Alle Einschätzungen sind Orientierungshilfen und ersetzen keine verbindliche anwaltliche Rechtsauskunft.

3.1 BGB-Grundlagen (§§ 398 ff.)

  • Formfreiheit: Abtretung nach § 398 BGB formlos gültig. Für die Plattform wird Textform (§ 126b BGB) empfohlen.
  • Bestimmtheitsgrundsatz: Fordify-Daten (Hauptforderung, Zinsen, Kosten, Datum) erfüllen das Erfordernis.
  • Abtretungsverbote (§ 399 BGB): pactum de non cedendo macht Abtretung unwirksam — Offenbarungspflicht in AGB.
  • Schuldnerbenachrichtigung (§ 409 BGB): Keine gesetzliche Pflicht, aber ohne Benachrichtigung kann Schuldner befreiend an Altgläubiger leisten (§ 407 BGB).

3.2 Rechtsdienstleistungsgesetz (RDG)

✅

Ergebnis: Keine RDG-Registrierung für Fordify erforderlich

Solange Fordify nur eine technische Plattform bereitstellt, keine Einschätzung zur Werthaltigkeit abgibt, keine Kaufempfehlung ausspricht und keine Vertretung der Parteien übernimmt, fällt der Betrieb nicht unter § 2 RDG. Käufer, die als Inkassounternehmen agieren, benötigen jedoch eine Registrierung nach § 10 RDG.

3.3 Regulierung: KWG und BaFin

und Preisstruktur

5.1 Das Design-Prinzip des Free Tiers

Das Free Tier muss Anwälte die Qualität von Fordify erleben lassen — aber im entscheidenden Moment (Export) einen Upgrade-Anreiz setzen. Der Anwalt hat 20 Minuten gearbeitet, sieht das perfekte Ergebnis — dann das Gate.

💡

Warum „kein Speichern + kein Export" und nicht „max. 3 Fälle"

Ein Fall-Limit frustriert früh und fühlt sich willkürlich an. Das Export-Gate trifft den Nutzer am emotionalsten Punkt — wenn die Arbeit fertig ist und er das Ergebnis verwenden möchte. Das erzeugt den stärksten Upgrade-Druck.

  • 29 €: „Weniger als eine Stunde Anwaltshonorar" — unter der psychologischen B2B-Kaufschwelle von 30 €.

7.2 Warum Paddle?

🏆

Paddle als Merchant of Record eliminiert alle USt-Pflichten

Paddle kauft das Produkt von Fordify und verkauft es weiter an den Endkunden. Damit liegt alle USt-Compliance (EU-weite USt, MOSS-Verfahren, Reverse Charge) bei Paddle. Fordify stellt nur eine Rechnung an Paddle — kein eigenes Steuer-Setup nötig.

Technische Architektur

8.1 Storage-Abstraktionsschicht (Free/Paid)

Die bestehende SPA verwendet localStorage an ~12 Stellen in app.js. Der Umbau führt eine Storage-Abstraktionsschicht ein:

TierStorage-BackendVerhalten
Free (kein Account)sessionStorageGeht beim Tab-Schließen verloren · überlebt F5
Paid (eingeloggt)localStorage + Supabase CloudPersistent · 2s debounced Cloud-Sync
 
const StorageBackend = {
  init(authState) {
    if (authState.isAuthenticated && authState.hasSubscription) {
      this._backend = localStorage;      // Paid: localStorage als Cache
      this._cloudSync = new CloudSync(); // + Supabase-Sync
    } else {
      this._backend = sessionStorage;    // Free: Session-only
    }
  },
  setItem(key, value) {
    this._backend.setItem(key, value);
    if (this._cloudSync) this._cloudSync.enqueue(key, value);
  }
};

Kapitel 9

DSGVO und Datenschutz

🔴

Statusänderung: Fordify wird Verantwortlicher

Aktuell verlassen keine Daten den Browser — Fordify ist kein DSGVO-Verantwortlicher. Mit dem Account-Modell und Supabase-Speicherung wird Fordify zum Verantwortlichen nach Art. 4 Nr. 7 DSGVO. Das ist eine fundamentale Statusänderung mit konkreten Pflichten.

9.1 Schuldnerdaten — Fordify als Auftragsverarbeiter

Schuldner-Namen und -Adressen in Falldaten sind personenbezogene Daten. Der Schuldner ist kein Vertragspartner von Fordify. Empfohlene Konstruktion:

Schuldner ← betroffen von → Anwalt (Verantwortlicher)
                                   ↓ erteilt Weisungen
                              Fordify (Auftragsverarbeiter)
                                   ↓ nutzt
                              Supabase (Sub-Auftragsverarbeiter)

Konsequenz: Fordify muss einen AVV mit jeder Kanzlei abschließen (Art. 28 DSGVO). Schuldnerdaten dürfen ausschließlich weisungsgebunden verwendet werden — kein KI-Training, keine Weitergabe.

9.2 Kritischste TOM: Row-Level Security

-- Jeder Nutzer sieht ausschließlich eigene Daten
CREATE POLICY "own_data" ON cases
  FOR ALL USING (auth.uid() = user_id);

CREATE POLICY "own_data" ON settings
  FOR ALL USING (auth.uid() = user_id);
⚠️

RLS muss vor Launch mit zwei Test-Accounts getestet sein

Ohne korrekte RLS-Konfiguration könnten Nutzer Daten anderer Nutzer sehen — das wäre eine DSGVO-Datenpanne mit Meldepflicht (Art. 33 DSGVO, 72-Stunden-Frist).

15.1 Blocker vor dem Freemium-Launch

Rechtlich / Vertraglich

  • AVV mit Supabase abgeschlossen und dokumentiert
  • Nutzungsbedingungen / AGB erstellt (inkl. AVV-Klauseln für Schuldnerdaten)
  • Datenschutzerklärung aktualisiert: Account, Supabase, Paddle als Drittanbieter
  • Paddle-DPA abgeschlossen
  • Verzeichnis von Verarbeitungstätigkeiten (VVT) angelegt

Technisch
